HLS就是用C++或者C实现RTL的方式

在使用HLS时，主要考虑两个方面

  • 设计的接口，即为顶层设计
  • 设计的功能，即为它所实现的算法

接口这里需要注意的是，不但需要描述port，同时接口也包括协议，对于port的位宽，类型以及方向是通过顶层文件的参数和返回值推断出来的，协议是通过端口的表现或者说行为表现出来的，接口类型包括：网线，寄存器，单向和双向握手信号，FIFO，存储器和总线

首先有一些常用的库

例如‘ap_cint.h’包括对于任意精度整数类型的支持

void led_twinkle(uint2 *led)

UG902

HLS有一些比较关键的阶段

  • scheduling(调度)：此种操作根据每个时钟周期决定哪些操作发生，而影响每个时钟周期内进行何种操作的因素包括

    • 时钟频率
    • 操作完成所需时间，由目标设备定义
    • 优化指令（这个在后面细说）

  • binding：确定哪个硬件资源实现预定操作

  • control logic extraction：创造一个有限状态机（FSM），对RTL设计中的操作进行排序

具体RTL实现了哪些操作呢：

  • 顶层函数综合为输入输出IO，有一点需要注意，顶层端口的数据位宽需要注意，防止计算数据溢出
  • C函数综合为RTL块，但是对于一个函数的子函数，所有子函数的例化都是采用一个RTL块实现的
  • C中的循环默认是未展开的，即串行时序，而不是并行，假设一次循环占用一个周期，那么一个从0到255的循环就要占用255个时间周期，显然并不满足利用FPGA的初衷，利用directive：unroll，可以使该255个循环以并行方式进行，但是这也意味着需要至少255个DSP，这时需要考虑你板子上的硬件资源
  • C代码中的arrays实现为块状RAM，如果array在顶层模块中，array实现为port，这些port可以访问设计之外的RAM实现临时存储

HLS创建implementation的根据是默认的行为，constrain，和优化指令

通过优化指令来修改和控制内部逻辑和IO端口的默认行为

那么利用C语言完成你的代码后，设计是否符合要求的判别需要看HLS生成的设计报告

  • Area：LUT，DSP,FF,RAM占用资源的多少
  • latency：完成整个操作的延时
  • loop iteration interval：函数可以接受新的输入数据之间的间隔时间
  • loop initiation interval：循环下次迭代开始之间间隔时间

接口优化

C语言的设计中，代码的输入输出都是通过形参完成，RTL输入输出主要通过port完成，并且需要遵循特定的端口协议进行操作，HLS提供两种IO协议

接口综合

当生成wraaper时，该函数的参数（或参数）被合成到RTL端口中。这个过程称为界面合成

#include "sum_io.h"
dout_t sum_io(din_t in1, din_t in2, dio_t *sum) &#123;
dout_t temp;
*sum = in1 + in2 + *sum;
temp = in1 + in2;
return  temp;

HLS可以综合的总线接口类型有三种

  • 时钟以及复位信号总线：ap_clk,ap_rst
  • 块语句接口：ap_start,ap_done,ap_ready,ap_idle
  • 端口级接口协议：in1，in2，sum_o

优化指令

优化指令应用对象有这些：

  • interface：此处接口起到类似ip的wrapper的作用，一般ip的配置接口都选择axi_lite总线进行配置，数据接口选择使用m_axis接口
  • 函数：将函数改成流水结构提高串行函数运行的速度等
  • 循环：将for循环等以流水线或者全展开的方式等进行并行计算
  • 数组：主要是制定RAM接口方式，决定一个周期内有多少个RAM可以提供多大位宽的数据接口
  • 区域：对括号内区域整体应用

优化指令的目标可以是指令文件也可以是源文件，如果选定directive，HLS将优化指令作为TCL命令插入solution文件夹下的directive.tcl文件夹中，如果选定源文件，优化指令以pragma的方式插入源文件中

HLS优化指令表
allocation指令

allocation用于指定函数实例的个数，比如函数中有四个函数的foo的实例，allowcation可以将实例数目限制在一个，四个实例都使用相同的RTL块来实现，

instances=：指定函数、运算符或内核的名称

limit=：可选地指定要在内核中使用的实例的限制

#pragma HLS allocation instances=  limit= 
#pragma HLS allocation instances=foo limit=2 function//将foo实例限制为2


void my_func(data_t angle) { 
#pragma HLS allocation instances=mul limit=1 operation

}//限制乘法器的操作数是1

HLS数组的优化

因为硬件中数组的实现hls往往采用单口或双口ram来实现，ram端口位宽往往是单次操作的数据的位宽，这样在循环累加的操作时，数据需要消耗大量时钟周期在数据读取上，每个数据需要进行取地址以及读数据两个周期

array partition数组展开

array partition展开有三种方式，block就是按照顺序将整个block拆开成多个ram，cyclic则类似多体交叉存储器的结构，这种结构利于实现i[0]+i[1]+i[2]之类的算法，complete就是完全拆开，有多少个ram就有多少个ram

dim主要用来进行高维数组的展开，绝定高维数组按照哪个维度进行展开

array map数组拼接

由于fpga的bram的块是由固定的大小的块组成，不同的数组若比较小，bram资源会存在大量浪费，可以使用array map将多个比较小的数组拼接成一个数组

#pragma HLS array_partition variable=m_accum  cyclic factor=2 dim=1
#pragma HLS array_partition variable=v_accum  cyclic factor=2 dim=1
#pragma HLS array_map variable=m_accum[0]  instance=mv_accum horizonal
#pragma HLS array_map variable=v_accum[0]  instance=mv_accum horizonal
#pragma HLS array_map variable=m_accum[1]  instance=mv_accum1 horizonal
#pragma HLS array_map variable=v_accum[1]  instance=mv_accum1 horizonal

接口的优化

interface指令
#pragma HLS interface mode= port= bundle= \
register register_mode= depth= offset= latency=\
clock= name= storage_type=\
num_read_outstanding= num_write_outstanding= \
max_read_burst_length= max_write_burst_length=

HLS的接口约束设置语言如上，下面介绍每个参数的作用

  • mode=

mode指定函数使用的函数参数的接口协议模式，模式可以指定为下面其一：

ap_none：不规定协议，单纯作为数据端口使用

ap_stable：也是作为数据端口使用，在reset信号后认为数据时钟稳定（稳定是否代表数据有效？）

ap_vld：作为valid信号使用

ap_ack：确认数据已读入或写入

m_axi：将所有端口实现为 AXI4 接口。

ap_none作用于形参，ap_none模式针对的是scalar inputs，ap_stable模式针对的是复位模式下配置输入发生改变的情况；
ap_ovld是针对 in_out 这种双向参数，ap_vld是输入有效标记信号；
ap_hs包括ap_vld和ap_ack，ap_vld表明当前输入或者输出是一个有效信号，ap_ack给出一个应答信号

由于以上接口在我的设计中不常用，故在此不再赘述，我设计接口通常采用axi-stream接口，具体接口时序参考博客：https://blog.csdn.net/weixin_42647783/article/details/102153098

  • port=

指定 INTERFACE pragma 适用的函数参数或函数返回的端口。

  • bundle=

默认情况下，HLS 工具将具有兼容选项的函数参数分组或捆绑到 RTL 代码中的接口端口中。 所有 AXI4-Lite (s_axilite) 接口都尽可能捆绑到一个 AXI4-Lite 端口中。 同样，默认情况下，所有指定为 AXI4 (m_axi) 接口的函数参数都捆绑到单个 AXI4 端口中。
具有兼容选项（例如模式、偏移和捆绑）的所有接口端口都被分组到一个接口端口中。 端口名称是从模式和捆绑包的组合中自动派生的，或者按照 -name 指定的方式命名。

在指定包名称时，应该使用所有小写字符。

  • register

寄存信号的标志位直到函数执行的最后一个周期，

  • register_mode=<forward|reverse|both|off>

此选项适用于 AXI4-Stream 接口，并指定寄存器是放置在正向路径（TDATA 和 TVALID）、反向路径（TREADY）、两条路径（TDATA、TVALID 和 TREADY）上，还是没有任何端口 信号将被注册（关闭）。 默认是both模式。 AXI4-Stream 侧通道信号被视为数据信号，两侧信号在TDATA被寄存时同时会被寄存。

  • depth=

需要注意的时depth是m_axis所必须的，通常大小为端口array的总数据数，比如a[8][9]被指定为m_axis格式，则深度为72

  • offset=

在 s_axilite 接口中， 指定寄存器映射中的地址。在 m_axi 接口中，此选项覆盖由 config_interface -m_axi_offset 选项指定的全局选项，并且 指定为：
off：不生成偏移端口。
direct：生成标量输入偏移端口。
slave：生成偏移端口并自动将其映射到 AXI4-Lite 从接口。 这是默认偏移量

一般选用slave即可，自动由AXI-Lite生成从接口

  • name=

指定端口名称

  • latency=

当模式为 m_axi 时，这指定了 AXI4 接口的预期延迟，允许设计在读取或写入预期之前的多个周期（延迟）启动总线请求。 如果这个数字太低，设计就会过早准备好并可能在等待总线时停止。 如果这个数字太高，可以授予总线访问权限，但总线可能会停止等待设计开始访问

HLS从外部存储器获得数据的快速方式

从DDR获取数据的方式主要通过AXI-HP接口的方式，但是如下接口HLS优化指令综合的电路只能是单次传输，而非burst传输，这样浪费大量时钟

void LeNet_CoConut(D32 img_in[28][28],D32 out[256]){
#pragma HLS INTERFACE s_axilite port=return bundle=CONTROL
#pragma HLS INTERFACE m_axi depth=256 port=out offset=slave bundle=OUT
#pragma HLS INTERFACE m_axi depth=784 port=img_in offset=slave bundle=IMG_IN
}

如果想让HP接口进行burst传输，需要选用memcpy函数，但需要注意的是，memcpy只能对综合电路的顶层函数传递时，参能完成综合成HP的burst传输模式

void example(volatile int *a){
#pragma HLS INTERFACE m_axi depth=50 port=a
#pragma HLS INTERFACE s_axilite port=return
//Port a is assigned to an AXI4 master interface
int i;
int buff[50];
//memcpy creates a burst access to memory
memcpy(buff,(const int*)a,50*sizeof(int));
for(i=0; i < 50; i++){
buff[i] = buff[i] + 100;
}
memcpy((int *)a,buff,50*sizeof(int));
}

两次memcpy实现了顶层模块与外部存储器的读写模式，其中以上例中的最后一条语句为例，burst的输出可以用for循环来实现

//1
for(i=0; i < 50; i++)&#123;
#pragma HLS PIPELINE
a[i] = buff[i];
&#125;


//2
memcpy((int *)a,buff,50*sizeof(int));

其中1和2都可以实现顶层的burst传输，但是使用for循环来进行burst传输有几个需要注意的点

  • 循环必须被流水线指令展开
  • 访问地址是升序
  • 没有条件语句
  • 嵌套的循环，不要展开内部的循环，不然会抑制突发传输

为了保证突发传输使得ip具有高吞吐量，推荐使用第二种方式

为了指定突发传输行为，有以下几个参数可以进行约束

1.latency：在前面已有介绍，在此不再赘述

2.max_read_burst_length：最大突发读长度

3.num_read_outstanding：在没有响应的情况下，发出读请求的最大次数，同时这个参数也与axisfifo的大小有关系
$$
fifo size = num_read_outstandingmax_read_burst_lengthword_size.
$$
4.max_write_burst_length

5.num_write_outstanding
